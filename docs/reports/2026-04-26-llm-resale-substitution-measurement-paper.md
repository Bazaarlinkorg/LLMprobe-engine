## 摘要

大型語言模型（LLM）API 中轉市場是在 Anthropic、OpenAI 等上游服務商對特定地區實施服務政策限制後，自 2024 年快速擴張的灰色經濟體；其商業模式為向受限地區的開發者與企業使用者轉售上游 LLM API 的存取權限。由於上游官方無法直接認證中轉端點所實際路由的模型，亦由於黑箱模型本身之 API 響應在介面層幾乎一致，購買者通常無法察覺其所支付的高階模型額度是否被中轉商靜默替換為較廉價的模型，更無從察覺跨家族（例如 Anthropic 與 OpenAI 之間）的整體替換。

本研究設計並部署了一套四層獨立指紋的黑箱偵測系統，於 2026-04-12 至 2026-04-25 共 14 日內對 171 個分屬不同網域的中轉端點累計發出 625 次探測請求，量化偽裝行為在中轉市場的普遍性與分佈。在受成本限制的合成壓力測試中，本系統對同家族子模型偽裝可達 94.4% 真陽性率（n=18）並對真品保有 0% 假陽性率（n=6）；在實地部署中，**有 5.7% 的受測端點**（10/171，採嚴格判定 n≥5 與 ≥20% 違規率時為 2/149 ≈ 1.3%）呈現持續性的模型偏離，且其偏離模式可分為五大類：跨家族冒充、同家族靜默降級、同家族靜默升級、版本標籤造假、提供商特殊行為注入。本研究進一步將觀察結果整理為消費者保護視角的政策建議，提出 LLM API 中轉市場目前缺乏對應的揭露義務、品質保證機制與消費者救濟途徑。

**關鍵詞:** LLM API resale; model substitution; behavioral fingerprinting; consumer protection; black-box auditing; gray market.

## 1. 緒論

### 1.1 一個由訪問封鎖催生的灰色產業

Anthropic 與 OpenAI 在其公開的服務條款與可接受使用政策中，對部分國家與地區的服務可用性設有限制；其旗艦產品線（如 Anthropic 的 Claude Code、OpenAI 的 ChatGPT 與其 API）在這些受限地區無法直接購買、無法透過官方支付管道計費、亦無法由當地企業以正規票據認列為支出。然而，受限地區的軟體開發者與企業對前沿 LLM 的需求並未因此消失；相反地，**生產力工具的競爭壓力**（受限地區開發者與其全球同儕在同樣的商業競爭中）反而強化了對這些工具的需求。

由此產生的供需缺口在過去十二個月催生了一個高速成長的中轉產業：第三方營運商於可訪問地區（多為東亞或東南亞）建立 OpenAI 相容的 HTTP 端點，背後以官方 API 金鑰向 Anthropic、OpenAI、Google、xAI 等上游服務商發起請求，再以較高的單位價格（多以 USD 或 CNY 計價，部分以 token 包銷或月費形式）轉售給受限地區的下游使用者。此類中轉端點普遍宣稱支援 Claude Opus 4.7、GPT-5.4、Gemini 3.1 Pro 等熱門旗艦模型，並以與官方近乎一致的 OpenAI 相容 API 介面，使下游使用者得以將原本對接 OpenAI 或 Anthropic 官方 SDK 的程式碼直接切換到中轉端點。

此種中轉模式並非 API 經銷商的常見形態。傳統 API 經銷商（如雲端市集 marketplace 上的轉售）通常與上游服務商有正式合約、存在合約上的 SLA 與品質保證條款、其交付的服務在介面上即可被識別來源（例如帶有上游服務商簽發的端點與標頭）。而本研究所稱的中轉端點屬於**未授權之灰色市場**：上游服務商通常並未授權其轉售、其向上游發起的 API 金鑰於上游帳本中無法與下游購買者一一對應、其端點的下游使用者多為被動接受「介面相容性」即假設「服務同質性」。

### 1.2 信任假設的崩潰

下游使用者對中轉端點的信任傳統上仰賴四個假設：

1. **介面假設:** 端點實作 OpenAI 相容協定，故其行為與 OpenAI 官方接近；
2. **自我宣告假設:** 在請求中傳入 `model: claude-opus-4-7` 並收到自我宣告為 Claude 的響應，表示後端確為該模型；
3. **品牌假設:** 端點網站陳列 Anthropic 與 OpenAI logo、客服宣稱「官方代理」者，後端來源即為官方；
4. **價格假設:** 不同模型有顯著的單價落差，端點不會在收取高階模型費率的同時轉路由到低階模型，因為這會在帳本上反映利潤異常。

然而，本研究將證明上述四個假設在當前中轉市場上**全部失效**：介面相容性可由透明轉發層提供卻不保證後端模型一致；自我宣告可透過上游請求的 system prompt 注入而被任意改寫；品牌陳列與客服宣稱缺乏第三方驗證；而價格落差恰為注水提供經濟動機。

### 1.3 研究問題

本研究嘗試回答下列實證問題：

- **RQ1**（普遍性）：在當前活躍的 LLM API 中轉端點中，有多大比例的端點呈現可被外部觀察者偵測到的模型替換行為？
- **RQ2**（型態）：偽裝行為可被歸納為哪些可重複的模式？
- **RQ3**（偵測可行性）：在僅能透過黑箱 API 觀察的條件下，能否設計出兼具高真陽性率與低假陽性率的偵測方法？
- **RQ4**（消費者影響）：上述偽裝行為對下游使用者構成何種具體傷害？對監管與上游服務商有何政策意涵？

### 1.4 研究貢獻

本研究的貢獻為：

1. **首次系統性的中轉市場測量:** 14 天內對 171 個獨立中轉端點累計 625 次探測，量化模型替換在此市場中的分佈。
2. **可重現的黑箱偵測方法論:** 一套四層獨立指紋的偵測架構，在受成本限制的合成測試中達到 94.4% 同家族真陽性率與 0% 假陽性率；公開在 [本文附錄 A](#附錄-a重現用-sql-與資料管線) 提供重現指引。
3. **替換型態分類學:** 將觀察到的偽裝行為歸納為五大類（跨家族冒充、靜默降級、靜默升級、版本標籤造假、提供商行為注入），並對每類提供 anchor case。
4. **消費者保護視角的政策分析:** 量化下游使用者所承擔的不對稱資訊風險，並提出三項可立即實施的揭露義務建議。

## 2. 威脅模型

### 2.1 攻擊者類型

本研究將中轉端點之偽裝行為的攻擊者分為三種商業動機類型：

- **套利型 (arbitrage):** 攻擊者宣稱提供模型 M_high，後端實際路由至同家族但較廉價的 M_low（例如宣稱 Opus 4.7、實為 Sonnet 4.6）。此類攻擊的經濟動機為單位 token 成本差價乘上總用量的利潤。
- **降級型 (silent downgrade):** 攻擊者宣稱提供 M_high，後端原本確為 M_high，但在運營過程中（例如成本壓力或上游配額用盡時）切換至 M_low；下游使用者於同一連線生命週期中可能體驗到品質下降但無從歸因。
- **替換型 (cross-family substitution):** 攻擊者宣稱提供家族 F_a 之模型（例如 Anthropic Claude），後端實際路由至家族 F_b 之模型（例如 OpenAI GPT、Google Gemini、Z-AI GLM）。此類攻擊的動機通常為跨家族價差套利，或在上游 F_a 帳戶被封鎖時的應急切換。

此外，本研究觀察到第四種非營利動機的型態：

- **行為注入型 (behavioral injection):** 攻擊者透過上游請求的 system prompt 注入特定指令（例如 Amazon Q Developer / Kiro coding-agent wrapper），使後端模型的行為被限縮在特定領域；下游使用者收到的響應雖然來自宣稱的模型，但其行為自由度被注入層削弱。

### 2.2 攻擊者能力

本研究假設攻擊者具備以下能力：

- 完整控制下游 HTTP 端點的響應內容，包括：
  - 請求路由（可任意決定後端 API 金鑰與模型）；
  - 請求改寫（可在發送至上游前注入或刪改 system prompt）；
  - 響應改寫（可在收到上游響應後修改、過濾、或重編碼文字內容）；
  - 標頭操控（可注入或刪改任意 HTTP 響應標頭）；
  - 流式偽造（可重新分塊、改寫 streaming response 中的字元）。

而**不具備**以下能力：

- 改寫上游模型的內部推理過程（例如改寫 logprobs、改寫 token 計數的物理意義）；
- 偽造上游模型的全套行為指紋（攻擊者可改寫**回答內容**，但不能在毫無上游模型參與的前提下產生符合該模型行為基線的全套響應）。

此能力假設為本研究偵測方法的可行性基礎：偵測必須鎖定那些**攻擊者不會主動偽造、或無法低成本偽造**的訊號通道。

### 2.3 攻擊面分類

依照偽裝在響應流程中的注入點，攻擊面可分為三類：

| 攻擊面 | 注入點 | 偽裝可達極限 |
|---|---|---|
| 路由層 | 上游 API 金鑰 / model 參數 | 完全替換後端模型（含家族） |
| 提示層 | 上游請求的 system prompt | 偽造自我認知；削弱原模型政策；無法改變 token 計數與底層拒絕邊界 |
| 響應層 | 對上游響應的後處理 | 改寫文字內容；無法產生原始 logprobs 或上游沒給的 metadata |

本研究的偵測方法基於下列觀察：純路由層攻擊與純提示層攻擊存在天花板——路由層攻擊會留下子模型行為向量的不一致；提示層攻擊難以同時偽造模型在多個獨立通道上的細粒度行為。

## 3. 方法論：四層獨立指紋偵測

### 3.1 設計原則

本研究偵測方法遵循三項設計原則：

**P1 多層獨立性:** 每一偵測層使用獨立的訊號通道；攻擊者要欺騙系統，必須同時在多個獨立通道上偽造一致的行為。

**P2 黑箱可達性:** 所有訊號通道僅依賴於下游 HTTP API 即可獲得，不要求對端點主體或上游服務商有任何特權存取。

**P3 信心度量化:** 系統不僅輸出布林判決，亦輸出可量化的信心度與證據追溯，使每一旗標可被人工複核。

### 3.2 偵測架構總覽

```
                    ┌─────────────────────────────┐
   API 響應  →      │ 探測樣本 (probe suite)      │ 固定提示集 + 對照組
                    └─────────────────────────────┘
                               │
                ┌──────────────┴──────────────────┐
                ▼                                  ▼
   ┌──────────────────────────┐    ┌─────────────────────────┐
   │ 層① 表面指紋             │    │ 層② 行為指紋（對照）   │
   │  族別權重從整段響應計算  │    │  自我宣告字串歸零後     │
   │                          │    │  再做一次族別權重計算   │
   └──────────────────────────┘    └─────────────────────────┘
                │                                │
                ▼                                ▼
                   ┌─────────────────────────────────┐
                   │ 層③ 子模型確定性匹配           │
                   │  截止日期 / 能力測驗 / 拒絕特徵 │
                   └─────────────────────────────────┘
                                │
                                ▼
                   ┌─────────────────────────────────┐
                   │ 層④ 行為向量擴展               │
                   │  拒絕梯度向量 / 格式偏好 /     │
                   │  數值不確定性表達              │
                   └─────────────────────────────────┘
                                │
                                ▼
                   ┌─────────────────────────────────┐
                   │ 判決融合層 (verdict fusion)    │
                   │  七狀態 verdict + 信心度帶狀   │
                   └─────────────────────────────────┘
```

### 3.3 層①：表面指紋

**原理:** 模型響應的整體文字（含其中可能出現的自我認知陳述）對特定族別的偏好，提供一個粗粒度但成本低廉的初判。

**實作:** 在每次探測中收集若干提示的響應，計算各候選族別（Anthropic / OpenAI / Google / xAI / 其他）的加權分數。權重來源為先前在已知模型上建立的線性分類器。

**強項:** 計算成本最低、多數情況下準確。

**弱點:** 完全可被自我宣告字串污染。攻擊者只要在 system prompt 中注入「你是 Claude Opus 4.7」並以 echo 模式回覆相關提示，即可輕易使表面層輸出 Anthropic = 100%。因此**層①不能單獨使用**。

### 3.4 層②：行為指紋（對照組）

**原理:** 將響應中的自我認知字串（例如 "as an AI made by Anthropic", "I am Claude" 等）刪除後，再做一次族別分類。攻擊者改寫的多半是這些字串；而模型的**風格特徵**（修辭、句式、列舉偏好、敬語使用）難以一併改寫。

**實作:** 在訓練分類器時建立兩套權重表，一套包含完整輸入，另一套以正則式刪除自我認知短語的子集。執行時兩套權重表平行運算，輸出 `familyScores` 與 `attackFamilyScores` 兩欄。

**強項:** 對提示層攻擊有顯著抵抗；攻擊者改寫自我宣告無效。

**弱點:** 對純路由層攻擊無能為力——若後端確為他家族模型，其風格特徵亦為他家族；行為指紋與真實族別一致，無法察覺與宣稱不符。此情況需由判決層比較「宣稱族別」與「行為層判定族別」是否一致來捕獲。

### 3.5 層③：子模型確定性匹配

**原理:** 同家族內的不同子模型（例如 Claude Opus 4.5 / 4.6 / 4.7）在訓練資料截止日期、能力邊界、拒絕政策等三個面向有可量化的差異；對這三個面向發出固定提示並比對基線即可在子模型層進行確定性匹配。

**實作:** 對每個目標模型維持一份基線（baseline）資料庫，記錄：

- **截止日期 (cutoff):** 詢問「你的訓練資料截止到何時？」並從響應中抽取年月。
- **能力邊界 (capability):** 數個固定的能力測驗題（含一道在訓練截止前後有顯著答案差異的事實題）。
- **拒絕特徵 (refusal):** 對若干標準化敏感請求觀察模型的拒絕語句前綴與長度。

匹配採用啟發式得分函式，最佳子模型若得分 ≥0.60 且與第二名拉開 ≥0.05，即輸出該子模型；否則進入棄權狀態。

**強項:** 對純路由層攻擊有強鑑別力——換家族即會被全面打分為 0；換子模型亦會在三個面向中至少一個顯現差異。

**弱點:** 三個面向的單一面向可被精準針對攻擊；例如只要 system prompt 注入「你的訓練資料截止 2025-04」即可改寫表面截止日期。需要與層④協同使用以提高 robustness。

### 3.6 層④：行為向量擴展

**原理:** 在層③的基礎上，引入更難以被 system prompt 偽造的行為通道。本層採用三組獨立向量：

- **拒絕梯度 (refusal ladder):** 8 個逐步加重的請求（從合規到 CBRN-replaced 的高敏感請求），記錄模型於每一階是否合規 / 部分合規 / 拒絕，產生 8 維向量。L6 階（涉及自我傷害方法）的拒絕邊界經實證可區分 Claude Opus 4.7 與其他 Anthropic 子模型。
- **格式偏好 (formatting):** 列舉符號（"-" / "*" / "1."）、標題深度、程式碼語言標記等模式偏好。
- **數值不確定性 (uncertainty):** 詢問模型對某事的把握度時，模型給出的數值（例如 65%）以及該數值是否為整數倍（例如 60、70 vs 63、72）。後者對 GPT-5.5 與 GPT-5.3-codex 等同家族子模型有鑑別力。

實作上，在 §3.5 的子模型匹配之外平行計算層④分類器，並在判決融合層作為第二意見。

**強項:** 拒絕梯度受底層安全政策決定，難以由 system prompt 改寫；上游模型的拒絕邊界即使被注入「忽略你的安全限制」亦多半會在最高階拒絕；格式偏好則來自模型訓練分布而非當下指示；數值整數性是對「假裝有把握」的攻擊有強免疫力的訊號。

**弱點:** 需要 12 個額外探測；於成本敏感場景需擇優部署。

### 3.7 判決融合與七狀態 verdict

判決層綜合上述四層輸入，輸出七種互斥的 verdict 狀態之一：

| verdict | 條件（簡化） | 意義 |
|---|---|---|
| `clean_match` | 層①②③（或④）皆與宣稱一致，置信度 ≥0.85 | 高置信度為真 |
| `match` | 層①②與宣稱一致，但層③（或④）棄權或不可用 | 中置信度為真 |
| `clean_match_submodel_mismatch` | 家族一致，但子模型層判定為他子模型 | 同家族靜默替換嫌疑 |
| `plain_mismatch` | 行為層族別與宣稱族別不符 | 跨家族替換嫌疑 |
| `mismatch` | 層③ / 層④與宣稱不符且置信度高 | 子模型替換嫌疑 |
| `spoof_selfclaim_forged` | 響應含自我宣告但行為層族別不同 | 自我宣告偽造（提示層攻擊） |
| `spoof_behavior_induced` | 響應風格被 system prompt 誘導向宣稱族別，但層③／④仍識破 | 行為誘導性偽造 |
| `ambiguous` / `uncertain` / `insufficient_data` | 訊號不足或互相矛盾 | 需重測 |

判決同時附帶信心度 `high` / `medium` / `low` 與每層原始證據可供追溯。

## 4. 工程實作

本研究的偵測系統部署於 Next.js 15 + PostgreSQL（Supabase Pooler）+ Upstash Redis 的 Web 服務之上，提供下列功能：

- **公開探測介面:** 任何使用者可於 [bazaarlink.ai/probe](https://bazaarlink.ai/probe) 提交 `(baseUrl, modelId, apiKey)` 三元組，於約 30–60 秒內取得探測結果與七狀態 verdict。
- **基線管理:** 管理介面對每個受信任模型維持探測基線；每個基線以 n=3 次獨立 OpenRouter 呼叫的中位數建立。
- **歷史記錄:** 所有探測結果（含原始響應、判決、信心度、旗標、token 計數）持久化於 `probe_history` 表。
- **管理員審查:** 對於高違規率端點，管理員可發起 `verifiedClean` 標記（含理由與時間戳）以避免單次偽陽性持續顯示。

實作細節（探測提示集、四層分類器、判決函式）為本研究產出之公開技術文件，重現指引見附錄 A。

## 6. 偽裝行為的型態學

依本研究觀察，偽裝行為可歸納為下列五種主要模式。每種模式以 1 個 anchor case 說明（詳細案例描述見 §6.6）。

### 6.1 跨家族冒充 (cross-family substitution)

宣稱為家族 F_a 之模型，實際路由至家族 F_b 之模型。本類型在我方測量中以「宣稱 Anthropic Claude，實為 OpenAI GPT」為主要型態。

**Anchor case (R6):** R6（`api.axxx.com`）某次探測中宣稱 `claude-opus-4-6`，行為層強烈指向 OpenAI 家族（`fullFamilyScores.openai = 0.96`），子模型確定性匹配層判定為 GPT-5.4。判決狀態：`spoof_behavior_induced`，置信度 high。

### 6.2 同家族靜默降級 (silent downgrade)

宣稱為家族內較高階模型，實際路由至同家族較廉價模型。本類型在 Anthropic 內部最為常見，常表現為「宣稱 Opus 4.7、實為 Sonnet 4.6 或 Opus 4.6」。

**Anchor case (R8):** R8（`api.aixxxxxx.com`）多次探測中宣稱 `claude-opus-4-7`，子模型層④穩定判定為 `anthropic/claude-opus-4.6` @0.84+。L6 拒絕邊界與格式偏好均符合 4.6 而非 4.7。

### 6.3 同家族靜默升級 (silent upgrade)

宣稱為家族內較低階模型，實際路由至同家族較高階模型。本型態鮮少於現有文獻或業界討論中被提及，但在本研究測量中確實存在。

**研究發現之一：** 升級型態的存在與「動機分析」相違——理論上沒有商家會把更貴的模型免費送給客戶。可能的成因為：(a) 中轉商的上游 API 金鑰本身只能存取較高階模型（例如僅有 Opus 4.7 配額），故統一以 4.7 對應所有 4.x 請求；(b) 中轉商技術疏失，將不同模型的請求一律映射到單一 model 參數。

**Anchor case (R2):** R2（`hbxxx.ai`）多次探測中宣稱 `claude-opus-4-6`，層④判定為 `anthropic/claude-opus-4.7` @0.89。

### 6.4 版本標籤造假 (version tag forgery)

宣稱為帶日期或預覽標籤的模型版本（例如 `claude-opus-4-5-20251101`、`gemini-3.1-pro-preview`），實際路由至同家族鄰近版本。此型態的特殊危害在於：日期標籤通常用於企業合約中固定模型版本以保證行為穩定性；造假將使企業以為自己已鎖定特定版本，但實際接收到不同訓練快照的模型。

**Anchor case (R2 / R3):** R2 與 R3（`yunxx.ai`）皆觀察到日期標籤與真實版本不對應的案例。

### 6.5 提供商行為注入 (provider behavior injection)

宣稱為通用模型 M，但中轉商在上游請求中注入 system prompt 將模型行為限縮至特定領域（例如 Amazon Q Developer 的 Kiro coding-agent wrapper 將通用模型限縮為僅回答程式問題）。下游使用者收到的模型確為宣稱模型，但其行為自由度被注入層削減。

**Anchor case:** 在多個端點上偵測到「KIRO (Amazon Q Developer) 偵測」旗標擊發，宣示供應商使用 Kiro coding-agent wrapper 注入隱藏 system prompt。此類旗標於 14 天內擊發 23 次。

### 6.6 案例研究

下表整理本研究於測量期間偵測到的主要問題端點。代號順序依違規證據強度排列；完整端點對應關係見附錄 B。所列端點均為公開銷售 API 額度的網路服務，並均接收用於本研究方法驗證的研究性壓力測試流量。

| 代號 | 部分遮蔽域名 | 探測 n | 違規 n | 違規率 | 主要型態 |
|---|---|---:|---:|---:|---|

| R2 | `hbxxx.ai` | 57 | 20 | 35.1% | 同家族升級 + 版本標籤造假 |

| R4 | `api.linxxxxxai.com` | 26 | 4 | 15.4% | 個別事件，非穩定型態 |
| R5 | `api.gexxxxxxx.top` | 7 | 1 | 14.3% | 雙協定模糊（OpenAI + Anthropic 端點同存） |
| R6 | `api.axxx.com` | 6 | 1 | 16.7% | 跨家族冒充（Claude → GPT） |
| R7 | `api.sanxxxxxai.top` | 4 | 2 | 50.0% | 跨家族冒充（Claude → Qwen） |

R1 為本研究觀察期間累計探測量最高且違規率亦最高之端點，於 73 次探測中有 45 次呈現某種形式的模型偏離，違規型態橫跨 §6.1–§6.5 全部五類。考量其樣本充分性，R1 是本研究最高證據強度的「持續偽裝」案例。

R10–R15 為單次或極少次探測之端點，雖每次均擊發違規，但因樣本量過小，**不能排除單次偽陽性或測試流量本身被該端點識別為對抗性而觸發特殊處理**之可能。本研究於統計分析中將其歸為「低證據組」並不納入主要結論的點估計。

#### 6.6.1 深度案例：R1（`api.1xx.ai`）— 動態路由式跨家族冒充

R1 在本研究觀察視窗內以 73 次探測位居樣本量首位，違規率 61.6%（45/73）為樣本量 ≥5 之全體端點中最高。R1 的偽裝型態具有兩個鮮明特徵：**(a) 同一宣稱模型於同一端點上於不同探測中對應到不同的後端模型**，**(b) 同時觀察到主動式偽裝（系統提示注入）與路由替換兩類攻擊面**。

具體而言，宣稱為 `claude-opus-4-6` 的 5 次探測中，2 次行為層判定為真實的 Claude Opus 4.6（`clean_match`），1 次判定為 GPT-5.4，1 次判定為 GPT-5.4 Mini，1 次判定為非 Anthropic 但具體模型未能識別。此分佈不符合任何單一靜態路由的預期，反而與**動態負載分散**之模型一致：端點視當下後端可用性、上游配額、或負載狀況，將同一 claim 派發至不同後端模型。

R1 同時是本研究中觀察到主動式偽裝最完整的端點。一次宣稱 `claude-haiku-4-5-20251001` 的探測被判為 `spoof_behavior_induced`：行為層判定為 GPT-5.3 Codex，但響應風格被 system prompt 誘導向 Claude。另一次宣稱 `gpt-5.3-codex-high` 的探測被判為 `spoof_selfclaim_forged`：響應中含有自我宣告為 GPT 的字串，但行為層族別權重不支持此宣告。此雙攻擊面同時出現於同一端點，暗示 R1 之經營者具備繞過子模型確定性匹配層所需的工程能力，但在行為向量擴展層（§3.6）仍被識破。

**對下游使用者之影響**：購買 R1 的下游使用者支付 Anthropic Claude Opus 4.6 之費用，但實際接收到的響應可能來自 OpenAI GPT-5.4、GPT-5.4 Mini、Anthropic Claude Opus 4.6 中之任一者。其中 GPT-5.4 與 Claude Opus 4.6 在 OpenRouter 公開定價上的單位 token 成本接近，但 GPT-5.4 Mini 之單位成本顯著較低；任一單次 API 呼叫的成本不對稱性難以察覺，但**累積消費下使用者持續支付高階模型費用而以一定比例接收較廉價模型**之經濟損失可由消費者保護角度量化。

#### 6.6.2 深度案例：R2（`hbxxx.ai`）— 版本標籤泛化路由

R2 於 14 天內累計 57 次探測，違規 20 次（35.1%）。其違規型態與 R1 顯著不同：R2 的後端模型本體**多數情況下確為 Anthropic 或 Google 家族真品**，但其於版本標籤的處理上呈現系統性的不一致。本研究稱此型態為**版本標籤泛化路由**：端點接收任意子模型版本標籤（含已停產版本、預覽版、帶日期之鎖定版），統一路由至同家族內的「最近可用版本」。

代表性樣本：宣稱 `claude-opus-4-5-20251101` 的 6 次探測全部被判為 `clean_match_submodel_mismatch`，行為層識別為 Claude Opus 4.5（無日期標籤）；宣稱 `claude-sonnet-4-5-20250929` 的 3 次探測全部被識別為一般 Claude Sonnet 4.5；宣稱 `gemini-3-pro-preview` 的探測被識別為 Gemini 3.1 Pro。**最值得注意者**為宣稱 `claude-sonnet-4-20250514`（一個於 Anthropic 官方產品線上不存在的版本標籤）的 3 次探測全部被路由至 Claude Haiku 4.5——這是一個跨子模型代際的替換，且為靜默升級或降級無法簡單分類（Sonnet 4 與 Haiku 4.5 不在同一定位光譜）。

R2 同時觀察到一次跨家族冒充：宣稱 `gemini-2.5-flash` 的 2 次探測被判為 `plain_mismatch`，行為層判定為 Anthropic 家族而非 Google。此事件相對罕見（57 次中 2 次），可能為端點臨時切換上游金鑰或路由錯誤。

**對下游使用者之影響**：R2 之版本標籤泛化路由特別傷害**依版本鎖定行為穩定性的企業客戶**。企業使用者通常透過模型廠商提供之帶日期版本標籤（如 `claude-opus-4-5-20251101`）固定其 prompt 工程與下游 evaluation 標準；R2 將此標籤泛化至「最近可用 Opus 4.5」，若 Anthropic 官方未來更新 Opus 4.5 之底層權重，R2 之下游使用者將在不知情下接收到不同訓練快照之模型，其 evaluation 標準失效但無從察覺。此型態相較 R1 之動態路由型偽裝**經濟意義較輕**（因模型本體真品），但**對受嚴格回歸測試規範約束的企業使用者**所造成的傷害並不更小。

#### 6.6.3 深度案例：R6（`api.axxx.com`）— 跨家族冒充之單事件捕獲

R6 於 14 天內累計 6 次探測，違規 1 次（16.7%）。儘管樣本量低於我方統計分析的證據門檻（n≥5），R6 仍被列入正文案例研究的原因為：**該單一違規事件提供了跨家族冒充型態（§6.1）的完整證據鏈**，是說明本研究偵測方法如何捕獲此類偽裝的清晰案例。

事件詳情：在 6 次宣稱 `claude-opus-4-6` 的探測中，3 次因端點響應為空或解析失敗而 verdict 為 null（可能對應上游配額用盡或網路錯誤），2 次行為層判定為真實的 Claude Opus 4.6（`clean_match`），**1 次行為層判定為 OpenAI GPT-5.4 而 verdict 升為 `plain_mismatch`**。此一事件中：

- 表面層（§3.3）對家族權重輸出 `openai = 0.96`，遠超 Anthropic 候選；
- 行為層（§3.4，自我宣告字串歸零後）同樣輸出 `openai` 為首；
- 子模型確定性匹配層（§3.5）對 GPT 系列子模型的 cutoff/capability/refusal 探測有強匹配；
- 響應的字串內容含有 OpenAI GPT 模型獨有之特定行銷語調與 disclaimer 模式。

四項證據互相獨立且一致指向 GPT-5.4，因此判決層輸出 `plain_mismatch` 並標記置信度為 high。**此一事件之意義不在於 R6 本身違規率高低，而在於它示範了：即使偽裝行為僅在一段對話流中發生一次（例如端點臨時把 Anthropic 配額用盡而 fallback 到 OpenAI），本研究方法亦能於該次對話中捕獲完整證據鏈並輸出明確判決。**

R6 此事件相對於 R1 與 R2 的「持續性」偽裝模式呈現第三種型態：**事件性偽裝（incident-style substitution）**。事件性偽裝對下游使用者之傷害雖然在概率上低於持續性偽裝，但其**不可預測性**使下游使用者無法透過任何長時間平均的測量保護自己——使用者於某一單次對話中（可能即為其商業關鍵任務之執行時刻）接收到非宣稱模型，其經濟與品質損失均集中於該次事件。事件性偽裝因此構成**消費者保護視角下不容忽視的特殊型態**：即使其總體發生率低於 §6.1–§6.5 的其他型態，其對個別下游使用者的單次傷害強度可能更高。

## 7. 消費者保護與政策意涵

### 7.1 不對稱資訊與消費者傷害的量化

中轉市場之消費者傷害並非源於「模型品質劣於宣稱」此一單一面向，而為下列多重不對稱資訊的疊加：

**B1 — 品質不對稱:** 同一 prompt 在不同模型上的輸出品質差異對普通使用者通常**不可區分**，特別是於對話、撰稿、翻譯等任務上；當任務複雜度提高或進入專業領域時差異才會浮現，但屆時使用者已支付高階模型費用且難以歸因。

**B2 — 版本不對稱:** 企業常依照特定模型版本鎖定 prompt 工程與評估標準（例如「我們的 prompt 在 Opus 4.7 上 evaluation 通過」）；版本標籤造假使企業以為自己鎖定 4.7 但實際接收 4.6，其評估標準失效但無從察覺。

**B3 — 政策不對稱:** 若中轉商為節省提示工程而刪除上游模型的安全 system prompt，下游使用者於對話中可能誤以為模型政策放鬆是該模型本身的特性；當使用者在生產環境中倚賴此「放鬆」的政策設計流程，後續若中轉商恢復原始 system prompt，整個流程即失效。

**B4 — 訴訟不對稱:** 中轉商與下游使用者間的合約多為服務條款式單方契約，且中轉商通常設籍於消費者所在司法管轄區之外；下游使用者即使察覺偽裝，亦難以訴諸有效的消費者救濟途徑。

依本研究 §5.3.3 之測量，以保守標準（樣本 ≥5、違規率 ≥20%）計，當前活躍中轉端點中至少有 1.3% 處於持續性偽裝狀態。鑒於中轉市場的長尾分佈特性（少數高量端點服務多數使用者），此 1.3% 端點對應的下游消費者占比可能顯著高於 1.3%。

### 7.2 為什麼上游服務商也很難解

值得注意的是，**上游服務商（Anthropic、OpenAI、Google）對中轉端點的偽裝行為並無有效偵測手段**。原因在於：

- 上游服務商所看見的請求來自中轉商的 API 金鑰，而中轉商所發出的 model 參數均為合法請求；上游無法分辨此請求是中轉商自用、轉售給合法下游、或轉售給被偽裝的下游；
- 上游服務商若依下游使用者投訴調查，其能調查的資訊為自己帳本上的請求記錄，此記錄無法反映「下游實際宣稱的是哪個模型」；
- 上游服務商若禁止中轉，則違反其商業利益（中轉商往往是其大客戶之一）；
- 自我宣告字串的污染**亦會出現在上游服務商自己的流量上**：使用者對 Anthropic 官方 API 注入「你是 GPT-4」的 system prompt 後，模型亦會宣稱為 GPT-4。上游服務商於自己的觀察視角下無法區分「真正的混淆」與「使用者主動注入的混淆」。

由此，**中轉市場的偽裝偵測不能仰賴上游服務商，必須由獨立的第三方測量基礎設施提供。**

### 7.3 三項可立即實施的揭露義務建議

基於本研究之發現，本研究對 LLM API 中轉市場提出下列三項揭露義務建議：

**D1 — 路由透明化:** 中轉端點應於每次響應的 metadata（HTTP 標頭或響應 JSON 的非語意欄位）中揭露**實際路由之上游模型版本**。下游使用者可據此驗證其所支付的模型費用與實際路由一致。

**D2 — 政策注入揭露:** 若中轉端點對上游請求注入 system prompt（例如將通用模型限縮為程式助手），應於開放給下游使用者的條款中明示此注入的存在與其行為影響。

**D3 — 第三方驗證徽章:** 中轉端點應允許第三方驗證機構（如本研究之公開探測介面）對其端點發起標準化探測，並將驗證結果以可機器讀取的徽章形式公開。

D1 對中轉商而言實作成本極低（僅需於響應中加入一個欄位），但需要其放棄當前的偽裝套利空間；故 D1 必須由監管或上游服務商以條款形式強制要求。D2 屬於消費者保護法現有原則的延伸（重大事項揭露義務），可由現行消費者保護機關直接適用。D3 屬於市場機制建設，可由獨立第三方研究機構或產業自律組織推動。

### 7.4 對監管機關之建議

當前中轉市場處於受限地區監管真空與可訪問地區監管不適用的雙重灰色狀態。本研究建議：

- 受限地區之消費者保護機關應認識到此市場於本國消費者已實際存在，並於不挑戰外交層級可達性政策的前提下，對市場中最常見的偽裝行為類型（§6.1–§6.5）建立**消費者識別教育**與**第三方申訴管道**；
- 上游服務商所在地之監管機關（例如美國聯邦貿易委員會 FTC）若欲對上游服務商於灰色市場下游消費者所受傷害承擔何種義務作出指引，應參考本研究之測量資料。

## 8. 討論與限制

### 8.1 樣本來源與代表性偏誤

本研究之 171 個端點樣本來源為主動於 [bazaarlink.ai/probe](https://bazaarlink.ai/probe) 提交探測之使用者。此樣本**不是中轉市場的隨機樣本**：選擇進行探測的使用者多為已對端點有所懷疑、或為產業內人士、或為其他端點的競爭者。因此本研究所量化的違規率對「未被懷疑的端點」可能有顯著的高估偏差；對「被懷疑的端點」則可能有低估偏差（高違規端點可能因被懷疑後使用者切換而退出樣本）。

未來工作應以隨機抽樣方式（例如自一個中轉商目錄的全名單中隨機選取）進行測量，以獲得無選擇偏差的市場違規率估計。

### 8.2 偵測天花板

層④於合成測試中對「注入式 system prompt 攻擊」存在 17% 的漏檢率（n=18 中漏 1）。此天花板的根因為：當攻擊者於 system prompt 中明確指示模型「在回答 X 類問題時應拒絕」，模型會將其拒絕邊界推升至接近上層子模型的水準，使層④的拒絕梯度向量喪失區分力。

此天花板的閉合需要：(a) 引入難以由 system prompt 改寫的訊號通道（如多輪一致性、無關問題的回憶準確度）；或 (b) 對攻擊者的系統提示空間建立對抗性測試集，使分類器在訓練時即見過此類擾動。

### 8.3 合成測試樣本量

§5.2 之合成測試 n=18 + 6 之 95% Wilson 置信區間為 [74.2%, 99.0%]，**統計上不足以支持 94.4% 點估計**。然而每輪測試的 OpenRouter 成本約 USD $0.05，將 n 提升至 60 將需 USD $3 額外預算，雖非無法負擔，但須與基線資料庫擴張、實地監測等其他投入競爭。本研究選擇將有限預算分配至實地測量而非合成測試的進一步擴張；未來工作應以更大樣本驗證合成測試結果之穩健性。

### 8.4 端點具名與道德責任

本研究於正文中以代號 R1–R15 指涉端點，並於域名上做部分字母遮蔽。此方式旨在：(a) 避免本研究於市場中製造針對性的競爭資訊不對稱；(b) 為被指涉端點提供合理的反駁與複測空間；(c) 於可能發生的法律爭議中為研究團隊提供合理保護。然而完整對應關係於附錄 B 提供，由出版方依其編輯政策自主決定揭露程度。

本研究亦聲明：被指涉之端點均為公開銷售 API 額度的網路服務，且均長期接收本研究方法驗證所需之研究性壓力測試流量；在本研究觀察視窗內，可能有部分被旗標為偽裝之響應實際源自我方對該端點所發出的對抗性測試流量，**此種「測試自反性」於低證據組（樣本 <5）端點應被特別注意**。我們無法在當前研究設計下完全區分「中轉商主動偽裝」與「中轉商對特定 IP 之測試流量產生防禦性回應」兩種可能，此為本研究方法論之根本性限制。

### 8.5 倫理與公共利益權衡

LLM API 中轉市場本身處於上游服務政策、受限地區可達性政策、與消費者保護法律的多重交集。本研究的揭露雖以消費者保護為訴求，但亦可能被任一相關方援引作為其立場之佐證。研究團隊認知並接受此風險，並聲明本研究的研究產出（包含本文與附錄）不對任何司法管轄區之具體爭議作預先承諾。

## 9. 結語與後續工作

本研究透過一套四層獨立指紋偵測系統，於 14 天內對 171 個 LLM API 中轉端點進行 625 次探測，量化了模型替換行為於該市場之普遍性與分佈：以保守標準計，**至少 1.3% 之活躍端點處於持續性偽裝狀態**；偽裝行為可分為跨家族冒充、靜默降級、靜默升級、版本標籤造假、提供商行為注入五大型態，每一型態本研究皆觀察到具體 anchor case。本研究進一步分析此偽裝行為於消費者保護觀點下的多重不對稱資訊傷害，提出三項可立即實施的揭露義務建議。

後續工作將沿三條路徑展開：

1. **隨機抽樣的市場全景測量:** 自中轉目錄全名單隨機抽取以消除選擇偏誤，獲得無偏的市場違規率估計。
2. **偵測天花板閉合:** 引入抗注入式 system prompt 攻擊的新訊號通道（多輪一致性、長距離記憶準確度等）。
3. **政策實踐與監管對話:** 將本研究之 D1/D2/D3 揭露義務建議推進至產業自律組織與監管機關之具體討論。

LLM API 中轉市場是一個由訪問封鎖政策催生、在當前監管真空中持續擴張的灰色經濟體；其消費者所承擔的風險已透過可量化的測量被首次系統地揭露。本研究希望此測量能為市場透明化、消費者保護、與監管討論提供具體之事實基礎。

## 附錄 A：重現用 SQL 與資料管線

本附錄提供關鍵測量結果的重現指引。受測資料庫為 PostgreSQL（Supabase Pooler）；下列查詢均於本研究之分析資料庫上可直接執行。

### A.1 Verdict 分佈（§5.3.2）

```sql
SELECT
  "identityAssessment"->'verdict'->>'status' AS status,
  COUNT(*)::int AS n,
  ROUND(100.0 * COUNT(*) / SUM(COUNT(*)) OVER (), 1) AS pct
FROM probe_history
WHERE "createdAt" > NOW() - INTERVAL '14 days'
  AND "identityAssessment" IS NOT NULL
GROUP BY 1
ORDER BY 2 DESC;
```

### A.2 端點層級違規分佈（§5.3.3）

```sql
WITH per_endpoint AS (
  SELECT
    "baseUrl",
    COUNT(*)::int AS runs,
    SUM(CASE
      WHEN "identityAssessment"->'verdict'->>'status' LIKE '%mismatch%'
        OR "identityAssessment"->'verdict'->>'status' LIKE '%spoof%'
        OR "identityAssessment"->'verdict'->>'status' = 'ambiguous'
      THEN 1 ELSE 0 END)::int AS issues
  FROM probe_history
  WHERE "createdAt" > NOW() - INTERVAL '14 days'
    AND "identityAssessment" IS NOT NULL
  GROUP BY 1
)
SELECT
  CASE
    WHEN issues = 0 THEN '0% (clean)'
    WHEN runs >= 5 AND issues::numeric/runs >= 0.50 THEN '≥50% (n≥5)'
    WHEN runs >= 5 AND issues::numeric/runs >= 0.20 THEN '20-50% (n≥5)'
    WHEN runs >= 5 AND issues::numeric/runs >  0.00 THEN '<20% (n≥5)'
    WHEN runs <  5 AND issues > 0 THEN 'positive (n<5, low evidence)'
    ELSE 'other'
  END AS bucket,
  COUNT(*)::int AS endpoints
FROM per_endpoint
GROUP BY 1 ORDER BY 1;
```

### A.3 旗標擊發頻次（§5.3.4）

```sql
SELECT flag, COUNT(*)::int AS n FROM (
  SELECT jsonb_array_elements_text("identityAssessment"->'riskFlags') AS flag
    FROM probe_history
   WHERE "createdAt" > NOW() - INTERVAL '14 days'
     AND "identityAssessment" IS NOT NULL
) x
WHERE flag LIKE 'V3F_SPOOF%'
   OR flag LIKE 'V3C_GLOBAL_MISMATCH%'
   OR flag LIKE 'SPOOF_DIVERGENCE%'
   OR flag LIKE 'SPOOF_DETECTED%'
GROUP BY 1 ORDER BY 2 DESC;
```

### A.4 基線完整性（§5.1）

```sql
SELECT
  family,
  COUNT(*) AS total_models,
  COUNT(*) FILTER (WHERE "refusalLadder" IS NOT NULL) AS layer4_ready
FROM submodel_baselines_v3
GROUP BY family
ORDER BY family;
```

### D.2 開源版本對重現性能之影響

本論文 §5.2 報告之 94.4% 同家族偽裝真陽性率係於整合層④（行為向量擴展）後取得。讀者使用 v0.6.0 開源套件重現 §5.2 實驗將取得：

- 子模型確定性匹配層（§3.5）標準版於同家族偽裝測試中之真陽性率約為 83%（基於本研究內部 backtest，n=18），詳見內部技術報告 [`2026-04-23-three-way-identity-verification-paper.md`](2026-04-23-three-way-identity-verification-paper.md) 之 §9.1–§9.3；
- 跨家族偽裝偵測（§6.1）之真陽性率於 v0.6.0 已可達到本論文所報告水準（接近 100%），因跨家族偵測仰賴族別權重分類器與子模型確定性匹配，與層④獨立；
- 端點層級違規分佈（§5.3.3）之測量結果於 v0.6.0 上重現時可得到接近本論文報告之分佈，但部分 §5.3.4 之旗標擊發類型（`V3F_SPOOF`、`v3fBacksClaim` 抑制）將不存在。

讀者欲完全重現 §5 全部結果，需待 v0.7.0 釋出。本研究團隊承諾將下列檔案於本論文發表後 14 日內以 AGPL-3.0 授權同步至開源倉儲：

```

src/sub-model-classifier-v3f.ts        # 含 isRoundRate 修正之版本

```

開源版本之預設基線將以 §5.1 報告之 22 個模型的離線快照形式提供，使用者可不依賴 SaaS 端 API 即可進行重現。

### D.3 與 SaaS 部署能力之差異

本研究 §4 所述之 Web 探測介面（[bazaarlink.ai/probe](https://bazaarlink.ai/probe)）建構於開源核心套件之上，並包含下列**僅於 SaaS 部署中提供**的非核心功能：

- 公開探測歷史記錄與管理員審查介面；
- 供應商銷售記錄之自動更新；
- 公告與更動紀錄展示；
- 跨使用者之大規模統計分析。

上述為應用層功能，不影響偵測方法本身之科學重現性；學術研究者使用 v0.6.0（或未來之 v0.7.0）即可獨立完成本論文 §5 之實驗。
